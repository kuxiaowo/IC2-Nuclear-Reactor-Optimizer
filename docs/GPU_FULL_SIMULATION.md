# CUDA 完整模拟穷举

## 结论

这个后端实现了“一个 CUDA 线程负责一个完整布局”。它不是静态近似评分，也不是只覆盖简单散热片的固定点证书：每个线程独立保存反应堆堆热、各槽组件类型、热量、耐久、首次临界、首次损坏、融毁、累计安全发电和周期历史。

界面中的 `CUDA 完整模拟（实验）` 会启用该路径。原来的 CUDA 固定点证书仍然保留，便于复现和比较。

## 为什么原先不能给复杂布局发“安全证书”

固定点证书只在少数实际状态转换后，完整热状态连续两次完全相同时提前结束。有限冷却单元、冷凝模块、换热器和有限耐久反射板会引入随时间变化的热量或耐久；仅凭某一时刻的净散热量，无法推出未来永远不损坏、不过热。对这类布局强行发证书会产生假阳性，因此旧后端只能返回“未知”并交给 CPU。

完整 GPU 模拟没有做这个推断。它直接执行时间序列，所以复杂布局不再需要固定点证书。

## 执行模型

1. CPU 将一批布局编码为紧凑组件编号并传到显存。
2. CUDA 网格中的第 `i` 个线程只读写第 `i` 个布局的状态。
3. 每个反应堆周期依次执行官方行优先热循环和发电循环，保留即时组件损坏副作用。
4. 默认每次 kernel 最多执行 256 个反应堆周期；未结束的线程在下一次启动继续。这样既保留线程内完整序列，也允许主机检查取消请求，并降低 Windows WDDM 长 kernel 超时风险。
5. 固定状态使用与 CPU 相同的逐字段比较和快进规则。一般周期状态只在 20,000 周期边界检查；64 位哈希仅作候选筛选，命中后仍逐字段比较，因此哈希碰撞不会造成错误的稳态判断。

线程之间没有状态共享。不同布局运行时长不同会造成 warp 分歧，这是该方案的主要性能代价；已经结束的线程会在后续分段启动时立即返回。

## 正确性验证

`tests/test_gpu_full_simulation.py` 包含：

- 所有组件家族的代表布局；
- 三列随机混合布局的短序列和 40,000 周期长序列；
- 54 槽布局；
- 普通穷举和 Mark I 两级穷举与 CPU 的完整计数、Mark、平均 EU/t、安全时长、安全裕量及排行榜差分；
- 通过完整穷举任务生成全局最优证明状态。

这些测试说明已覆盖样本与 CPU 一致，不等于形式化证明 CUDA 内核永远无缺陷。当前仍将该后端标为实验性；若修改 CPU 规则实现，应同步增加差分用例。

## 本机基准

在 RTX 5070 Ti、40,000 反应堆周期上预热 CUDA 后测试：

| 空间 | 布局数 | CPU | CUDA 证书 + 回退 | CUDA 完整模拟 | 结果 |
|---|---:|---:|---:|---:|---|
| 1 单铀 + 最多 1 个 10k 冷却单元 | 324 | 6.04 s | 6.07 s | 0.172 s | 三者排行榜一致 |
| 1 单铀 + 最多 3 个反应堆散热片 | 15,012 | 1.14 s | 0.600 s | 0.375 s | 三者排行榜一致 |

数值会受驱动、首次 JIT、后台图形负载和批大小影响。可用下列命令复现；脚本会先预热再计时：

```powershell
conda run --no-capture-output -n ic2-reactor-optimizer python scripts/benchmark_gpu_exhaustive.py --component coolant_10k --vent-cap 1
conda run --no-capture-output -n ic2-reactor-optimizer python scripts/benchmark_gpu_exhaustive.py --component reactor_heat_vent --vent-cap 3
```

当前数据中，完整模拟在两类空间都更快，尤其避免了复杂布局“GPU 判定后再 CPU 回退”的双重开销。但固定点证书后端更小、更容易审计，仍适合作为稳定默认值和交叉校验基线。
