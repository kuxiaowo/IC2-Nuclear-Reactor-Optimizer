# 大批量 GPU 计算后端设计边界

当前版本仍以 `ReactorSimulator` 为权威参考实现。`kernel_abi.py` 固定搜索器与
并行内核之间的数据契约；`ScalarPackedEvaluator` 是差分金标，
`NumbaPackedEvaluator` 提供 CPU 批量并行，`CudaPackedEvaluator` 已提供第一版
一线程一布局的 NVIDIA GPU 实现。CUDA 只替换计算执行器，不修改规则代码、
搜索计数或证明语义。

## 1. 固定输入 ABI

- 布局：C 连续 `uint8[batch, 6 * columns]`，数值码由
  `COMPONENT_ID_BY_CODE` 固定。
- 初始堆热：`int32[batch]`。
- 组件参数：结构分离数组，热量和耐久使用 `int32`，组件种类和燃料棒数使用
  `uint8`。
- 邻接：`int16[slots, 4]`，严格保持左、右、上、下顺序，缺失邻居填 `-1`。

ABI 不接受 Python 字符串、Pydantic 模型、动态字典或事件对象。CPU/GPU 后端
只能返回 `PackedEvaluationBatch` 中定义的固定宽度摘要；需要展示的字符串和
完整事件由 CPU 层按需重建。

## 2. 并行模型

布局之间完全独立，是主要并行维度。单个布局内部仍必须串行执行：

1. 槽位按行优先执行热阶段；
2. 槽位按相同顺序执行发电阶段；
3. 元件移除立即生效；
4. 邻居严格按左、右、上、下访问。

第一版 CUDA 内核采用“一线程处理一个布局”。同一布局的槽位没有无约束并行，
因为后者会破坏 IC2 的即时写回语义。

## 3. 调度结构

所有穷举后端都使用多个 CPU 生产进程并行枚举和剪枝，按 CPU 数动态增加中心
槽位分片以减少尾部空闲；Numba 在每个穷举进程内使用一条计算线程，避免嵌套
线程池过量并发。CUDA 另有一个独立 worker 持有唯一设备 context，而不是让
每个 `ProcessPoolExecutor` 子进程分别建立：

```text
多个 CPU 进程：骨架/冷却生成与数学剪枝
            ↓ sequence id
固定大小布局批次（建议从 4K–64K 实测）
            ↓
单 GPU worker / 持久 CUDA context
            ↓ 紧凑结果批次
CPU Mark 过滤、镜像归组、排行榜与证明计数
```

批次必须携带单调 sequence id，允许 GPU 按完成顺序返回而不改变布局计数。
取消任务时，尚未提交和未返回的批次都必须保留为未决，不能计入数学排除。

有限基础周期后仍未分类的布局使用独立续算上限。CPU 层先用当前平均 EU/t
上界与各目标 Mark 的第十名下界比较，只把仍有机会进入榜单的行送去续算。
GPU worker 因而只需支持“给定布局批次和统一周期上限”的纯函数接口，不承担
排行榜判断。未决布局的最大功率上界必须返回证明层，不能静默当作不可行。

当前混合调度基于 RTX 5070 Ti / i9-14900K 实测：64 和 1,024 个长布局时 CUDA
启动与单 lane 串行成本明显高于 Numba；8,192 个布局开始反超，16,384 个布局
约为 32 线程 Numba 的 3 倍。因此搜索器只把至少 8,192 行、且不超过 40,000
反应堆周期的批次交给 CUDA。小批次与长续算自动走 Numba；CUDA 编译、驱动、
显存或执行异常也会整批重算，绝不把失败结果计作不可行或数学剪枝。

## 4. 精确性要求

GPU 结果若参与 `proven_global`，必须与标量参考逐字段一致：

- 禁止 `fastmath`；
- 换热器比例、阈值和舍入必须转换为经证明等价的整数/有理数运算，或严格复现
  当前双精度路径；
- 热量、耐久、堆容量和事件周期不得使用会溢出的窄整数；
- 固定点和周期判断必须无哈希碰撞；
- GPU 异常、超时、溢出或未知结果必须回退 CPU，不能当作不可行。

## 5. 验证门槛

实现 GPU 后端前后必须保留以下测试层次：

1. `ScalarPackedEvaluator` 与现有 `evaluate_layout` 的固定金标；
2. 随机布局 Scalar/Numba/GPU 全摘要差分；
3. 换热、立即移除、有限反射板和容量边界的定向差分；
4. 小库存全枚举比较 `checked/evaluated/pruned/unresolved`、排行榜和证明标志；
5. 长时间先升温后稳定布局，防止把有限边界未分类误作永久不可行。

当前定向边界、固定随机布局、最大仓宽、长时间 SUC、搜索摘要和失败回退测试
均已逐字段通过，因此 CUDA 结果可以进入全局证明路径。后续若改变内核状态布局、
换热舍入或分段调度，仍必须重新通过同一组门槛。
